# Memory protocol

The agent reads three persistent layers, each with one job. Together they replace the "throw everything into a vector store and hope" approach with a deterministic, auditable retrieval model.

## The three layers

### 1. Hot pinboard — `~/.claude/projects/<project-id>/memory/MEMORY.md`

**Job:** Always loaded into the agent's context. ≤80 lines.

**Contains:**
- User identity (family, role, employer)
- Sticky preferences (writing voice, auto-approve, PDF defaults)
- Decision-time principles inlined as one-liners (5–8 entries)
- Pointers to the 5–8 currently active ventures with paths to their vault `_learnings.md`
- A pointer to `INDEX.md`

**Promotion rule:** an entry earns the hot pinboard if it must fire on every relevant task across multiple future sessions. Otherwise it lives in `INDEX.md`.

### 2. Cold index — `~/.claude/projects/<project-id>/memory/INDEX.md`

**Job:** Read on demand when the hot pinboard doesn't answer the question.

**Contains:**
- Locations (pointers to canonical files outside the memory dir)
- Pipelines (system docs, cron descriptions, MCP gotchas)
- Dormant ventures (auto-populated by the active-venture-refresh cron)
- Feedback / preferences that don't fire on every task

**Promotion rule:** new memory files default here. They only migrate to `MEMORY.md` after enough sessions prove they should fire automatically.

### 3. Per-venture state — `Vault/<path-to-venture>/_learnings.md`

**Job:** Read first when the task is venture-scoped. Append-only.

**Contains, per file:**
- Status snapshot (current positioning, headline facts)
- `## Learnings` — append-only insights, each with `*(from [[session-id]])*` source citation
- `## Key Decisions` — table of dated decisions with rationale and session link
- `## Open Threads` — unresolved items; move to Learnings or Decisions when resolved
- `## Related Sessions` — list of sessions that touched this domain

**Hard rule:** never rewrite. Always append. If a learning contradicts an existing one, flag the contradiction inline rather than silently replacing.

## Memory directory layout

The memory directory is partitioned into typed subfolders:

```
~/.claude/projects/<project-id>/memory/
├── MEMORY.md          # Hot pinboard, always loaded, ≤80 lines
├── INDEX.md           # Cold pointers, read on demand
├── user/              # kind:identity (Derek, family, body)
├── feedback/          # kind:feedback (corrections, confirmations, preferences)
├── principle/         # kind:principle (decision-time heuristics)
├── location/          # kind:location (pointers to canonical files)
└── pipeline/          # kind:pipeline (crons, MCPs, hooks, routing protocols)
```

Filenames mirror the subfolder: `user/user_*.md`, `feedback/feedback_*.md`, etc. Frontmatter `tags:` includes a matching `kind:` value.

**Note:** venture state does NOT live in the memory directory. It lives in the vault `_learnings.md` for that venture. Memory directory is for cross-session patterns (feedback, principles, locations, pipelines, identity), not project state.

## Memory file frontmatter

Each memory file declares its kind and decay rate:

```yaml
---
name: <kebab-case-slug>
description: <one-line summary>
type: reference
tags: [kind:principle, decay:low]      # or kind:feedback / kind:location / etc.
---
```

The minimum tag set: `kind:<x>` (which subfolder) + `decay:<y>` (low / medium / high). Add `venture:<slug>` or `domain:<slug>` when the entry is scoped to one.

## Search-first protocol

When the agent starts a session:

1. **Always-load:** read `~/.claude/CLAUDE.md` (user instructions) + `MEMORY.md` (hot pinboard). Cached after first read.
2. **Time-sensitive task?** Read `Vault/Today.md` (regenerated by cron 4x/day with calendar + due tasks + recent activity).
3. **Venture-scoped task?** Read the matching `_learnings.md`. Use [`pipeline_knowledge_router.md`](../skills/distill/learnings-resurface/) as a starting hint, not a directive.
4. **Cross-cutting or thin?** Grep across vault for related context — sibling `_learnings.md`, recent voice memos, session archives.
5. **Iterate.** Refine search terms based on what surfaces. Treat the first lookup as a hint, not the answer.

The Letta benchmark on agent memory found that filesystem + iteration beats specialized retrieval. Trust the iteration loop.

## Cost model

- **First read** of `MEMORY.md` in a session: ~$0.04 (uncached).
- **Subsequent reads** within 5 minutes: ~$0.004 (Anthropic prompt-cache hit).
- **Per-venture `_learnings.md` read**: $0.001–0.01 depending on size.
- **Grep across vault**: ~$0.002 in tokens.

Cost is a non-issue inside the envelope. Don't over-economize — the cache is generous and `grep` is cheap.

## What goes where — decision flowchart

```
Is this about Derek's identity, family, body, preferences?
  YES → memory/user/ or memory/feedback/
  NO  ↓

Is this a decision-time heuristic that should fire on every relevant task?
  YES → memory/principle/ + pointer in MEMORY.md hot pinboard
  NO  ↓

Is this a pointer to a canonical file or external resource?
  YES → memory/location/ + pointer in INDEX.md
  NO  ↓

Is this a system / cron / MCP / hook protocol document?
  YES → memory/pipeline/ + pointer in INDEX.md
  NO  ↓

Is this state for a specific venture?
  YES → that venture's Vault/<path>/_learnings.md (under ## Learnings or ## Key Decisions)
  NO  ↓

Is this an ephemeral working note for today?
  YES → Vault/Today.md (regenerated by cron, don't edit by hand)
  NO  ↓

You're probably trying to write something that doesn't have a home yet.
Stop and decide: new memory category, new venture folder, or out-of-scope.
```

## The cron jobs that maintain this

See [`automation/`](../automation/):

- **`weekly-memory-maintenance.sh`** — Sunday 11pm PT wrapper.
- **`dormancy-decay.py`** — marks `_learnings.md` as `status:dormant` after 90 days no touch.
- **`active-venture-refresh.py`** — demotes stale pinboard entries to `INDEX.md`.
- **`prune-memory-dryrun.py`** — audits the memory dir against the conventions; logs to `Vault/log.md`.
- **`learnings-resurface.sh`** — Sunday 10pm cross-corpus reconciliation.

The cron jobs read this protocol and enforce it. Without them, the conventions decay within a quarter.
